    SITUAÇÃO ATUAL:
    - [CONCLUÍDO] Opção B — Ordens manuais:

        pronta para copiar e enviar no Discord.

        Botão "🚀 Deploy" por fase com checkbox por operação, barra de progresso,
        badge de progresso no painel de fases e botão "📋 Copiar pendentes".
    - [PAUSADO] Opção A — Deploy automático no jogo.

    OPÇÃO A — Deploy automático (pausado, bloqueios técnicos):

      BLOQUEIO 1 — Autenticação EA por jogador:
        Tentativa de implementar fluxo OAuth (PKCE) com JUNO_PC_CLIENT.
        Problema: JUNO_PC_CLIENT só aceita redirect_uri=qrc:// (URI do EA App
        desktop), que não funciona em navegador web. A EA bloqueia envio do
        e-mail de 2FA quando o request vem de IP de servidor (Cloudflare Worker).
        O cliente EADOTCOM-WEB-SERVER tem redirect_uri web mas é confidencial
        (exige client_secret que não temos).
        → Retomar se a EA publicar um cliente OAuth público com redirect web,
          ou se for possível capturar o token do EA App instalado no PC.

      BLOQUEIO 2 — Endpoint de escrita do jogo (não documentado):
        A CG não publica API de escrita para TB. Para descobrir:
        - Interceptar tráfego do app oficial (Fiddler/mitmproxy) durante
          alocação manual de platoon → identificar URL, headers, payload.
        - Risco: endpoint pode mudar com atualizações.
        → Retomar após resolver o Bloqueio 1.

      BLOQUEIO 3 — Controle de permissão:
        Apenas liderança pode executar deploy. Requer lista de allycodes
        autorizados no KV e verificação no Worker.
        → Implementação simples após resolver os bloqueios 1 e 2.

    NOTA: worker/index.js e ui/authPanel.js já contêm a infraestrutura base
    (endpoints /auth/popup-init, /auth/callback, /auth/status no Worker;
    renderEAAuthSection() no frontend). Pronto para retomar quando os bloqueios
    acima forem resolvidos.
